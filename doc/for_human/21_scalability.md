# 확장성 — 사용자가 많아지면?

> **관련:** `doc/for_agent/01_architecture.md` · `doc/for_human/13_deployment.md`

## 비유

- **작은 카페** = 단일 Supabase 프로젝트 (현재)  
- **프랜차이즈** = 캐시, CDN, DB 인덱스, 모니터링  

## Supabase가 도와주는 것

- 사용자 수 늘면 **플랜 업그레이드**  
- Connection pooling, 읽기 복제 (유료 기능)  

## 프론트

- 정적 파일은 Vercel CDN으로 전 세계 배포  
- 실시간 채팅은 Supabase Realtime 채널 수 증가에 따른 비용 확인  

## 코드 구조

- **수업(course) 단위**로 데이터 나눔 → 한 수업이 커져도 다른 수업에 영향 적음  
- 역할(professor/student) 분리 → 기능 추가 시 권한만 확장  

## 우리가 당장 신경 쓸 것

1. DB 테이블 설계를 **처음부터 정규화** (중복 최소)  
2. RLS로 “전체 학생 데이터 한 번에 조회” 방지  
3. 파일은 Storage에, DB에는 **URL만**  

## 한계

대형 포털(수만 동시 접속) 수준은 **별도 아키텍처**가 필요합니다.  
대학 수업 단위 서비스에는 현재 스택으로 충분한 경우가 많습니다.
